# 차별화 통찰과 설계 함의

## 1. 로그의 통찰 구조화

```
일반 인식:
"AI가 리서치 잘하잖아 → 검색하면 되지"

실제 현실:
AI 서치 ≠ 좋은 자료
계획 없는 키워드 검색으로는 진짜 답이 안 나옴

시장 갭:
바이브 코더들이 구현은 잘하는데
첫 단추(소스)가 부실해서 결과 품질이 낮음

리서치팀의 차별화:
1. 계획 기반 수집 (무엇을, 왜 찾는지 명확)
2. API 연동 (일반 검색으로 닿지 못하는 자료원)
3. 진짜 좋은 자료 → 프로젝트 맞춤 답
```

작업 흐름의 첫 단계가 부실하면 그 뒤 모든 단계가 흔들린다는 통찰. 이건 시장에서 검증 가능한 페인포인트.

---

## 2. 설계 함의: Source와 Extract의 비대칭

지금까지 두 시스템을 50:50 대칭으로 다뤘지만, 이 통찰을 적용하면 비대칭입니다.

| 시스템 | 위상 | 설계 원칙 |
|---|---|---|
| **Source 시스템** | 플러그인의 본체. 차별화의 핵심 | 압도적으로 깊고 정교해야 함 |
| **Extract 시스템** | Source 가치를 실현하는 도구 | 견고하고 정직하면 됨 |

"좋은 입력만 들어오면 좋은 출력은 보장"이라는 비대칭.
Extract가 아무리 화려해도 입력이 부실하면 결과는 흔들림.

---

## 3. Source 시스템의 핵심 자산 (차별화 본체)

| 자산 | 역할 | 차별화 강도 |
|---|---|---|
| **API 연동 카탈로그** | DART, KOSIS, Naver, 도메인별 API | ★★★ |
| **도메인별 수집 프로파일** | 시장/제품/콘텐츠/경쟁사 등 유형별 전략 | ★★★ |
| **계획 단계 정교함** | search_plan.md의 품질이 결과를 결정 | ★★★ |
| **자가 학습 로그** | 어떤 자료원이 어떤 주제에 강한지 누적 | ★★ |
| **소스 다양성 강제** | 뉴스만 X. 1차/2차/리뷰/규제/통계 균형 | ★★ |
| **출처 검증** | 신뢰도/최신성/편향 평가 | ★★ |

이 6개가 Source 시스템의 진짜 알맹이.
일반 AI 리서치 도구가 갖지 못한 것들.

---

## 4. Extract 시스템의 역할 (안정성 우선)

| 자산 | 역할 |
|---|---|
| Handoff 강제 로드 | Source가 넘긴 맥락을 빠짐없이 흡수 |
| Analysis Question 설계 | 좋은 입력에서 좋은 질문 도출 |
| NotebookLM 분석 | notebook_query 정교한 운용 |
| Evidence 검증 | 반대 증거, 모순 점검 |
| 판단 가능한 형태로 출력 | 정보 X, 판단 근거 O |

Extract는 새롭지 않다. 견고하면 된다.

---

## 5. 플러그인 배포 관점에서

| 마케팅 메시지 후보 | 평가 |
|---|---|
| "AI 리서치 자동화" | ❌ 흔함. 차별화 없음 |
| "두 시스템 분리 리서치" | △ 구조 자랑, 가치 불명 |
| **"바이브 코딩의 첫 단추를 채워주는 리서치"** | ◎ 페인포인트 직격 |
| **"API 기반 진짜 자료 수집 + 판단 추출"** | ◎ 구체적, 차별화 명확 |

플러그인 명칭/포지셔닝도 Source의 강함을 드러내야 함.
"잘 만든 리서치 도구"가 아니라 "다른 데서 못 얻는 자료를 얻는 도구"로.
