# Source 시스템 재설계: 도메인 특화 + 갭 안내

## 1. 통찰 확장

```
사실:
목적이 달라지면 필요한 자료가 완전히 달라진다.

예시:
- 앱 개발자 → 앱 시장, 사용자 행동, 경쟁 앱
- 변리사 → 특허청 API, Google Patents, 국제 특허 DB
- 의사 → 의료 공공 데이터, 임상 가이드라인
- 부동산 → 국토부, 부동산원, KB시세

한계:
모든 도메인 API를 플러그인에 미리 통합할 수 없다.

해법:
1. 핵심 API는 통합
2. 도메인별 API는 프로파일로 확장
3. 누락된 자료는 "안내 보고서"로 제시
```

---

## 2. Source 시스템 3-Layer 구조

```
┌───────────────────────────────────────────────────┐
│ Layer 1: Core (모두 공통)                         │
│ - 범용 공공 API (KOSIS, 공공데이터포털, DART)     │
│ - 뉴스/학술 검색 (Naver, Google Scholar)          │
│ - Playwright fallback                             │
│ - 일반 웹 검색                                    │
└───────────────────────────────────────────────────┘
                       ↓
┌───────────────────────────────────────────────────┐
│ Layer 2: Domain Profile (선택 적용)               │
│ ┌──────────────┐ ┌──────────────┐ ┌────────────┐  │
│ │ 변리사       │ │ 의사         │ │ 앱 개발자  │  │
│ │ - 특허청 API │ │ - HIRA API   │ │ - 앱스토어 │  │
│ │ - Google     │ │ - 보건복지부 │ │ - SensorT  │  │
│ │   Patents    │ │ - 임상 DB    │ │ - 리뷰 수집│  │
│ └──────────────┘ └──────────────┘ └────────────┘  │
│  + 사용자가 추가 가능한 커스텀 프로파일           │
└───────────────────────────────────────────────────┘
                       ↓
┌───────────────────────────────────────────────────┐
│ Layer 3: Gap Analyzer (자동 갭 식별)              │
│ - 수집 결과 분석                                  │
│ - 누락 자료 영역 식별                             │
│ - "부족 자료 보고서" 생성                         │
│   ├── 어떤 자료가 부족한가                        │
│   ├── 어디서 얻을 수 있는가                       │
│   ├── API 발급 절차                               │
│   └── 대안 방법 (수동/외주/연구)                  │
└───────────────────────────────────────────────────┘
```

---

## 3. 각 Layer의 역할

### Layer 1: Core API

플러그인 설치 시 즉시 사용 가능한 기본 자산.
**한국 시장 + 범용 비즈니스**에 필요한 최소 세트.

| API | 용도 | 우선순위 |
|---|---|---|
| 공공데이터포털 | 정부 통계/공공 데이터 게이트웨이 | ★★★ |
| KOSIS | 통계청 데이터 | ★★★ |
| DART | 기업 공시 | ★★★ |
| Naver Search Ads | 키워드/검색량 | ★★ |
| 한국언론진흥재단 빅카인즈 | 뉴스 통합 | ★★ |
| Google Scholar | 학술 | ★★ |

이건 로그가 이미 일부 작업 이력 있는 자산. 재활용 가능.

### Layer 2: Domain Profile

**플러그인의 진짜 차별화 자산.**
도메인별로 미리 정의된 수집 전략 + API 세트.

각 프로파일 구성:
- 해당 도메인의 핵심 자료원 목록
- 자료원별 API 정보 (있으면 자동, 없으면 안내)
- 도메인 특화 검색 전략
- 수집 기준 (무엇을 신뢰할 만한 1차 자료로 볼 것인가)

배포 시 포함할 초기 프로파일 후보:

| 프로파일 | 주요 자료원 | 시장 크기 |
|---|---|---|
| 변리사/특허 | 특허청, Google Patents, WIPO | 중 |
| 의사/의료 | HIRA, 보건복지부, PubMed | 중 |
| 앱 개발자 | 앱스토어, SensorTower, 리뷰 | 대 |
| 부동산 | 국토부, 부동산원, KB시세 | 대 |
| 콘텐츠 크리에이터 | YouTube API, 트렌드, 키워드 | 대 |
| 1인 기업가/마케터 | Naver, 통계청, 소상공인진흥공단 | 대 |

**사용자 추가 메커니즘 필수**:
- 프로파일 정의 파일 (YAML/JSON) 표준 제공
- 커뮤니티에서 프로파일 공유 가능
- 이게 플러그인의 생태계 확장 동력

### Layer 3: Gap Analyzer (가장 독창적)

**이게 진짜 새로운 기능.**
다른 리서치 도구가 안 하는 것.

수집 후 자동 점검:
```
1. 수집된 자료 카테고리 분석
2. 목적 대비 누락 영역 식별
3. 부족 자료 보고서 생성
```

부족 자료 보고서 예시:

```markdown
## 부족 자료 보고서

### 수집된 자료
✅ 시장 규모 (KOSIS)
✅ 경쟁사 공시 (DART)
✅ 뉴스 동향 (빅카인즈)

### 부족한 자료
❌ 해외 특허 출원 동향
   → 권장: Google Patents API
   → 발급: 무료, 즉시
   → 안내: https://patents.google.com/api

❌ 임상 시험 데이터
   → 권장: ClinicalTrials.gov
   → 발급: 무료, 회원가입
   → 안내: https://clinicaltrials.gov

❌ 글로벌 시장 규모 (확정 데이터)
   → API 없음
   → 대안 1: Statista 유료 구독
   → 대안 2: 컨설팅 보고서 구매
   → 대안 3: 학술 논문 메타분석
```

이게 있으면:
- 사용자는 "어디서 자료를 더 얻을 수 있는지" 알게 됨
- 플러그인이 단순 수집 도구가 아니라 **리서치 가이드** 역할
- 모르는 자료원을 발견해주는 도구

## 4. 마케팅/배포 관점에서

도메인 프로파일이 마케팅 메시지의 핵심.

```
"변리사라면 특허 리서치 프로파일이 자동 작동합니다"
"의사라면 의료 공공 데이터를 자동 수집합니다"
"앱 개발자라면 시장 리서치가 한 번에 끝납니다"
```

각 도메인에 직접 어필 가능. 일반 "AI 리서치 도구"와 차원이 다른 포지셔닝.

플러그인 확장 모델:
- **무료**: Core + 1~2개 기본 프로파일
- **유료/프리미엄**: 모든 도메인 프로파일 + 갭 분석 심화
- **커뮤니티**: 사용자가 만든 프로파일 마켓플레이스

## 5. 핵심 차별화 한 줄

```
일반 리서치 도구:
"검색해서 모아드립니다"

NEXUS 리서치 플러그인:
"당신 도메인에 맞춰 진짜 자료를 모으고,
 부족한 부분이 어디인지까지 알려드립니다"
```

## 6. 결정이 필요한 항목

| 번호 | 항목 |
|---|---|
| P-1 | 초기 배포 시 포함할 도메인 프로파일 (몇 개? 어떤 것?) |
| P-2 | 사용자 추가 프로파일 표준 포맷 (YAML? JSON? Markdown+frontmatter?) |
| P-3 | Gap Analyzer의 자료원 DB는 어디에 둘 것인가 (플러그인 내장? 별도 repo?) |
| P-4 | 무료/유료 구분이 있는지, 전체 오픈소스인지 |
| P-5 | 로그 자신의 도메인(콘텐츠/AI 에이전시) 프로파일을 1번 프로파일로 만들어 dogfooding할지 |
