# 검색 플랜 작성 가이드

표준 양식(`templates/search_plan_template.md`) 각 섹션을 어떻게 채우는지에 대한 실용 규칙.
좋은 예 / 나쁜 예를 같이 둔다. **Plan 작성 30분 이내**를 목표로.

---

## 제1원칙: 사실관계 추정 금지 — 단일 마커 `[?]`

이 시스템은 "작성자의 학습 데이터 기억"이 아니라 "NotebookLM에 적재된 외부 자료"를 1단계 검증으로 삼는다. Plan 작성자가 자기 기억에서 사실을 단정하면 그 단정이 양식에 박혀 Source 팀이 그 값을 기준으로 검색하게 되고, 시스템 설계가 무너진다.

### 단일 마커

직접 1차 출처를 확인하지 못한 사실값(버전·통계·날짜·고유명사의 상태·분류 체계 등)에는 모두 `[?]`를 부착한다.

| 상황 | 표기 |
|---|---|
| 작성 중 모름 | `Expo SDK [?]` |
| 작성 중 임시 추정으로 적은 값 | `Expo SDK 53 [?]` (값과 함께 마커) |
| 사후 마킹 (이미 단정해버린 문장) | `... InfoQ Mobile은 RN 전문 아님 [?] 검증 안 함: 일반적 인상` |

이 한 종류로 압축. 다른 마커(`[모름]`, `[검증 필요]`, `TBD` 등) 혼용 금지.

### 사유 카테고리 5종 (정식)

마커에 사유를 붙일 땐 다음 5종 중 하나로 통일. 자유 서술 금지. 9-5 사후 검증에서 위반 패턴을 분류·집계해야 다음 Plan 양식 개선의 입력으로 쓸 수 있다.

| 사유 코드 | 의미 | 전형적 발생 자리 |
|---|---|---|
| `[?]` (사유 생략) | 단순 모름. 30초 확인 시도조차 안 한 상태 | 빠른 작성 중 임시 |
| `[?] LLM 추정` | LLM(Claude 등) 작성자가 training data 회상으로 적었으나 1차 확인 안 함 | 버전·날짜·통계 값 자리 전반 |
| `[?] 30초 룰 위반` | 30초 내 확인 가능했을 가능성 높으나 작성자가 건너뜀. 시간 여유 시 우선 해소 대상 | 공식 docs 한 클릭으로 확인 가능한 버전·기본값 |
| `[?] 검증 안 함: 일반적 인상` | 자료원 성격·시장 인식·평판 등을 막연한 인상으로 단정. 1차 확인 가능 여부 불명 | 자료원 적합성 평가, 시장 위상 단정 |
| `[?] 분류 체계 전제 미확인` | 분류 체계 자체(예: "Bare workflow vs Dev Build")가 현 시점 공식 분류와 일치한다는 전제를 확인 안 함 | 핵심 질문에서 분류로 비교를 요구할 때 |

### 카테고리 사용 규칙

- 사유는 한 마커당 **1개만**. 복수 해당 시 가장 강한 것 하나만 (우선순위: `분류 체계 전제 미확인` > `LLM 추정` > `검증 안 함: 일반적 인상` > `30초 룰 위반` > 사유 생략)
- 인간 작성자가 적은 값이라도 출처 확인 없이 일반 상식·인상에 기댄 경우 → `검증 안 함: 일반적 인상`
- LLM이 적은 값은 거의 항상 `LLM 추정` (예외: 그 자리에서 WebSearch/WebFetch 호출 후 URL 인라인 명시한 경우)
- 사유 텍스트는 **위 5종 문구를 그대로 복붙**. "약간 모름", "확신 없음" 같은 변형 금지 — 9-5 집계가 깨짐

### 30초 룰

사실값을 적기 전 자문: **"1차 출처로 직접 확인이 30초 안에 가능한가?"**

- 가능 → 확인하고 마커 없이 적음. URL을 옆에 인라인으로 메모해두면 9-5(사후 검증) 단계에서 빠름
- 불가능 → `[?]` 부착하고 진행. Plan 작성 30분 목표를 사실 한 줄에 5분 쏟느라 깨지 말 것

판단 기준 예시:
- 공식 docs 페이지 한 번 클릭으로 확인되는 stable 버전 → 30초 안. 확인
- 시장 점유율·매출 등 다중 출처 종합 필요한 값 → 30초 초과. `[?]`로 두고 Source 단계에 위임

### LLM 작성자 특례

Plan 보조 작성자가 LLM(Claude 등)인 경우, **LLM의 training data 회상은 "직접 확인"으로 간주하지 않는다**. cutoff·hallucination·버전 드리프트가 인간 작성자보다 더 심하기 때문.

LLM이 마커 없이 사실값을 적을 자격을 가지려면:
- 작성 도중 WebSearch/WebFetch/Context7/공식 docs URL을 **그 자리에서 호출**해 확인했고
- 호출한 출처 URL을 인라인으로 명시함

이 조건을 안 만족하면 모두 `[?]`. "내 학습 데이터엔 X였다"는 마커 없이 적을 자격 없음.

평가(검토) 단계에서도 동일: LLM 평가자가 단정으로 적은 자료원 성격·시장 인식 등에 1차 출처 확인이 없으면 retrospective `[?]`로 표시한다 (대표 사례: `evaluations/[2026-05-23][Mobile] React Native와 Expo 입문 학습 정리_평가.md`의 자료원 성격 단정 3건).

**Why**: 자체 AI 검색을 1단계 검증으로 쓰지 않고 외부 자료 기반 검증을 1단계로 쓰는 게 이 시스템의 존재 이유. 작성자가 LLM이라는 사실이 이 원칙을 약화시키지 않음. 오히려 강화시킴.

## 섹션 1: 메타

**규칙**: 주제는 한 줄에 끝나야 한다. 한 줄에 안 들어가면 범위가 넓다는 신호.

| 예 | 평가 |
|---|---|
| "AI 도구" | ❌ 너무 넓음. 키워드 폭주 확정 |
| "생성형 AI" | ❌ 시점·지역·관점 없음 |
| "2026년 한국 변리사 업무에 생성형 AI 도입 현황과 한계" | ✅ 시점·지역·산업·관점 명시 |

**목적 적기**: "이해하기 위해서"는 목적이 아니다.
- "X를 결정하기 위해", "Y 보고서 근거로", "Z 글의 인용 소스로"
- 학습 정리 목적이면 "다음에 어떤 토픽을 깊이 팔지 정하기 위해" 같이 후속 결정 단위로.

**마감 / 긴급도**: 선택 항목. 단독 학습이나 사전 조사 같은 비협업 작업이면 비워둬도 됨.

---

## 섹션 2: 목적 분해

### 2-3 "알 필요 없는 것"이 가장 중요

이유: 범위가 안 정해지면 키워드가 폭주한다. 무엇을 **안 볼지**를 정해야 무엇을 **볼지**가 명확해진다.
**1개 이상 필수.**

예 (주제: "한국 1인 SaaS 시장 진입 가능성"):
- 알 필요 없는 것:
  - 미국·유럽 SaaS 시장 전반
  - 엔터프라이즈(50인 이상) SaaS
  - 10년 이전의 historical 사례

이 3줄이 있으면 "Salesforce 시장 동향" 같은 노이즈가 제거된다.

## 섹션 3: 핵심 질문

**규칙**: 모든 질문이 "수집한 자료로 답할 수 있는가" 테스트를 통과해야 한다.

| 질문 | 평가 |
|---|---|
| "1인 SaaS는 성공할까?" | ❌ 미래 예측. 자료로 답 불가 |
| "1인 SaaS가 좋은 길인가?" | ❌ 가치 판단. 자료로 답 불가 |
| "최근 24개월간 한국에서 MRR 1만 달러를 넘긴 1인 SaaS는 몇 건이며, 공통 패턴은?" | ✅ 자료로 검증 가능 |
| "글로벌 1인 SaaS 창업자가 사용한 결제·인프라 스택의 빈도 분포는?" | ✅ 자료로 검증 가능 |

**질문 수**: 5~10개. 3개 이하면 깊이가 안 나오고, 10개 초과면 NotebookLM 응답이 산만해진다.

## 섹션 4: 키워드 체계

### 핵심 규칙: 원어 그대로

실제 검색이 이루어지는 언어로 원어 그대로 적는다. 번역해서 적지 말 것.

| 검색 언어 | 적는 방식 | 예 |
|---|---|---|
| 한국어 검색 | 한국어 원어 | `1인 SaaS`, `마이크로 SaaS`, `독립 개발자 수익화` |
| 영어 검색 | 영어 원어 | `solo founder SaaS`, `bootstrapped micro-SaaS`, `indie hacker MRR` |
| 중국어 검색 | 간체/번체 원어 | `独立开发者 SaaS`, `一人公司`, `微型 SaaS` |
| 일본어 검색 | 일본어 원어 | `一人 SaaS`, `個人開発 サブスク` |

### 한국어 주제라도 영어 최소 3개

이유: 1차 자료(공식 docs, 학술 논문, 글로벌 API 문서)는 영어가 압도적이다.
한국어만 모으면 1차 자료원이 비고 T2(뉴스)에 쏠린다.

### 추가 언어 (4-3)

중국어·일본어 등은 **주제 특성상 필요할 때만** 추가. 매번 "해당 없음" 적는 게 아니라, 필요 없으면 표를 통째로 삭제해도 OK.

추가 기준:
- 중국 시장·제조·기술 주제 → ZH 필수
- 일본 산업·문화·만화 주제 → JA 필수
- 그 외엔 KO + EN만으로 충분

### Primary / Secondary / Long-tail 구분

- **Primary**: 가장 정확한 핵심어. 3~5개.
- **Secondary**: 동의어, 유사어, 상위·하위 개념. 5~10개.
- **Long-tail**: 시나리오·질문형. "1인 SaaS MRR 1만 달러 달성 사례" 같이 길고 구체적.

### 제외 키워드

노이즈 차단용. 주제별로 자주 섞이는 다른 의미를 명시 제거.
예: 주제 "솔로 SaaS"인데 채용 광고가 다수 → 제외: `채용`, `구인`, `recruiting`

## 섹션 5: 소스 매트릭스

### "Google"은 자료원이 아니다 — 단, `site:` 검색식은 허용

| 표기 | 평가 |
|---|---|
| "Google에서 검색" | ❌ |
| "구글" | ❌ |
| "Google Scholar (scholar.google.com), `solo founder SaaS revenue` 검색" | ✅ |
| `Google 검색: site:velog.io "React Native" 입문` | ✅ (검색식이 자료원을 구체화) |
| `Google 검색: site:reddit.com "Expo Router migration"` | ✅ |

검색식이 단일 사이트·도메인을 지정하면 그 자체가 구체 자료원으로 인정된다.

### 묶음 자료원 표기

성격이 유사한 작은 자료원 여러 개는 **한 행으로 묶어도 됨**. 단 각 자료원이 명확히 식별 가능해야 한다.

| 표기 | 평가 |
|---|---|
| "한국 블로그 여러 개" | ❌ 식별 불가 |
| `한국 테크 블로그 (카카오·우아한형제들·토스): tech.kakao.com / techblog.woowahan.com / toss.tech (각 사이트 내 검색 "React Native")` | ✅ |
| `한국 개인 블로그 (Velog·Tistory): Google 검색 site:velog.io / site:tistory.com + "Expo 시작하기"` | ✅ |

### Tier 분리 규칙

| Tier | 정의 | 대표 예 |
|---|---|---|
| T1 (1차) | 자료 자체가 원본 | 공식 docs, 정부 통계(KOSIS, 공공데이터포털), 기업 공시(DART), 원 논문, 공식 API |
| T2 (2차) | 1차를 가공·분석 | 산업 리포트, 학술 리뷰, 신뢰 매체(빅카인즈, FT, NYT) 분석 기사 |
| T3 (3차) | 현장 의견·여론 | 커뮤니티(Reddit, HN, 디시·뽐뿌), SNS(X, 링크드인), 영상 자막(YouTube), 개인 블로그 |

### 가장 흔한 실패

뉴스(T2)에만 잔뜩 모으는 패턴. 뉴스만으로는 1차 사실도 현장 감각도 안 잡힌다.
**T1 ≥ 2개 + T3 최소 1개**가 안전 하한.

### 자료원별 필수 메타

| 항목 | 왜 필요한가 |
|---|---|
| URL / 검색식 | Source 팀이 헤매지 않게 |
| 검색 방식 | API인지 웹 UI인지 미리 알아야 인증 준비 |
| 인증 | 없으면 즉시 차단됨 |
| 파일 형식 | NotebookLM 적재 가능한가 사전 확인 |
| 목표 건수 | 끝없는 수집 방지 |
| 언어 | Tier 비율과 함께 다양성 계산 |
| **API 필요 여부** | Source 팀이 작업 시작 전 키 발급·도구 설치 사전 준비. 9-2(사후 보고)를 사전 명시로 전환 |

### `API 필요 여부` 컬럼 기록 규칙

| 표기 | 의미 |
|---|---|
| `불필요` | 웹 UI에서 URL 직접 add로 끝 |
| `[API 필요: GitHub Releases RSS/API]` | API 종류 명기 |
| `[API 필요: youtube-transcript-api / yt-dlp]` | 사용 도구 명기 (대체재 병기 OK) |
| `[? Source 팀 결정]` | 작성자가 판단 안 됨. 30초 룰로 판단 불가하면 마커 부착 |

목적: Source 팀이 자료원 표를 열자마자 "GitHub 토큰 발급해야겠다", "yt-dlp 설치 필요" 같은 사전 준비를 끝내고 본 작업에 들어가게 함.

## 섹션 6: 검색 파라미터

### 언어 가중치 결정 가이드

| 주제 성격 | 권장 가중치 |
|---|---|
| 한국 시장·법규 한정 | KO 70% / EN 25% / 기타 5% |
| 글로벌 기술·오픈소스 | KO 25% / EN 70% / 기타 5% |
| 중국 제조·시장 포함 | KO 30% / EN 40% / ZH 30% |
| 일본 만화·산업 포함 | KO 30% / EN 40% / JA 30% |

### 최신성 기본값

| 주제 성격 | 최신성 |
|---|---|
| 법규·규제·정책 | 6개월 이내 |
| 시장·경쟁 동향 | 6~12개월 |
| 기술 베스트 프랙티스 | 1~2년 |
| 라이브러리·API 사용법 | 6개월~1년 (버전 빠름) |
| 기초 개념·이론 | 무관 |

너무 짧으면 기반 자료가 비고, 너무 길면 낡은 사례가 섞인다.

### 언어 가중치와 Tier 비율의 상호작용

영어 우세(70%+)이면서 T1 비율(공식 docs)이 높으면 KO는 자연스럽게 T2·T3에 집중된다. 충돌이 아니라 정상 결과. 다만 **KO 자료가 모두 T3(개인 블로그)에만 있으면 위험** — KO T2(테크 블로그·매체)도 최소 1개는 확보.

## 섹션 7: NotebookLM 실행 지시

**규칙**: Source 팀이 모르는 내부 약어/맥락 금지. 그대로 복붙 수준으로 실행 가능해야 한다.

| 표기 | 평가 |
|---|---|
| "기존 노트북에 이어서 추가" | ❌ 어떤 노트북인지 모름 |
| "신규 노트북 생성, 이름: `[2026-05-23][SaaS] 한국 1인 SaaS 진입 가능성`" | ✅ |

**소스 추가 순서가 중요한 이유**: NotebookLM은 추가된 소스 전체를 컨텍스트로 활용하지만, 원전(T1)이 먼저 적재돼 있어야 분석(T2)·여론(T3)이 기반 위에 얹힌다. T3부터 넣으면 톤이 여론에 휘둘림.

## 섹션 9: Gap Report

**규칙**: 검색 끝나야 채워지는 섹션이지만, **Plan 단계에서 빈 자리를 미리 만들어 두는 것** 자체가 핵심.

이게 자가 진화 루프의 입구. Gap Report에 누적된 "필요했던 자료원"이 도메인 프로파일로 승격되면 다음 플랜이 더 강해진다. (참고: `reference/self_evolving_system.md`)

### 9-5: Plan 사실관계 검증 결과 (제1원칙 루프백)

Plan 본문에 들어간 작성자 단정(마커 없는 사실값)이 NotebookLM 적재 자료와 일치했는지를 사후 점검. 제1원칙(추정 금지) 위반이 어디서 일어났는지 누적해서 다음 Plan 작성 시 같은 자리에서 반복하지 않게 한다.

기록 항목:
- 통과 / 불일치 / `[?]` 마킹된 항목 각각 몇 건
- 불일치가 발생한 위치 (예: 7-3 메타 라벨링 예시, 핵심 질문 4번의 분류 등)
- 패턴 (예: "버전 정보를 자체 회상으로 단정하는 경향" → 다음 Plan에서 버전 컬럼은 기본 `[?]`로 시작)

대표 사례 (2026-05-23): RN/Expo 입문 학습 정리 플랜의 7-3 메타 라벨링이 `Expo SDK 53`(실제 56), `RN 0.77`(실제 0.85)로 1년 outdated. 9-5에 기록되어 이후 양식의 마커 규칙 강화로 이어짐.
